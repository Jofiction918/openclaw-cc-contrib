# memory-sorting — 衣柜式记忆整理

定期整理 `MEMORY.md` 和 `memory/*.md`，像整理衣柜一样：检测重复、过时、冲突内容，生成提案等你批准，批准后执行修改，保持记忆系统整洁有序。

## 何时使用

- 手动触发：`/memory-sorting`，定期整理（如每周一次）
- 当出现以下情况时主动建议：
  - 新增了重要项目约定，但不知道该放哪个文件
  - 发现不同文件对同一事情说法不一致
  - 某些 daily 文件已经过时但不确定能否删除
  - MEMORY.md 超过 200 行

## 执行步骤

### Step 1 — 全量扫描

扫描以下所有记忆文件：
```
MEMORY.md                              ← 长记忆层（核心）
memory/YYYY-MM-DD.md                   ← Daily 层（原始日志）
memory/ENTRYPATH.md                    ← 主题索引（如果存在）
```

按时间线汇总所有记忆条目，标记每条记忆的：
- **来源文件**（MEMORY.md 或某个 daily）
- **出现次数**（判断是否重复）
- **最后更新时间**（判断是否过时）
- **置信度**（高：用户明确说过 / 中：推断 / 低：模糊记忆）

### Step 2 — 问题检测

| 问题类型 | 检测逻辑 | 输出标记 |
|----------|---------|---------|
| 重复（Duplicate） | 同一内容在 ≥2 个文件中出现 | 🔁 |
| 过时（Stale） | 旧结论被新 daily 推翻；相对时间描述（"昨天"）存在超过7天 | ⏰ |
| 冲突（Conflict） | 同一问题不同文件说法矛盾 | ⚡ |
| 归属模糊（Ambiguous） | 分不清是项目规范还是个人偏好 | ❓ |
| 孤儿条目（Orphan） | 某 daily 中的条目从未被引用或提炼到 MEMORY.md | 🔍 |
| 碎片化（Fragmented） | 一个完整主题分散在 ≥3 个不同文件中 | 📜 |

### Step 3 — 生成提案报告

提案分三类，**每一项都必须包含**：
- 具体内容（旧/新对比）
- 依据（哪个文件的哪一行）
- 执行后的预期效果

```markdown
## 🗃️ 记忆整理提案 · {{YYYY-MM-DD}}

**提案生成时间**：{{HH:MM:SS}}
**扫描范围**：MEMORY.md + {{N}}个 daily 文件
**检测结果**：🔁 {{N}}条重复 | ⏰ {{N}}条过时 | ⚡ {{N}}条冲突 | ❓ {{N}}条待确认

---
```

### Step 4 — 执行

收到批准后，按以下顺序执行：

```
1. 执行所有 Promote（先写 MEMORY.md，建立新记忆）
2. 执行所有 Cleanup（删除/合并/更新）
3. 更新 ENTRYPATH.md（如果涉及文件增删）
4. 验证修改后的文件可读
5. 输出执行报告
```

## 检测例外（不会标记为问题）

- AI 资讯来源列表（不同 daily 出现相同 URL 是正常现象）
- 完全相同的工具名记录（如"安装了 skill X"，多次出现但上下文不同）
- 时间戳不同但内容相似的 TODO 条目（视为不同时间点的独立记录）

## 权限要求

- `FileRead`：读取所有 memory 文件
- `FileWrite` / `FileEdit`：修改 MEMORY.md 和 daily 文件
- `sessions_spawn`：（可选）fork 子 Agent 并行分析大量文件
