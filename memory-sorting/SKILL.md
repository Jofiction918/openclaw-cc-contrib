# memory-sorting — 衣柜式记忆整理

定期整理 `MEMORY.md` 和 `memory/*.md`，像整理衣柜一样：检测重复、过时、冲突内容，生成提案等你批准，批准后执行修改，保持记忆系统整洁有序。

## 何时使用

- 手动触发：`/memory-sorting`，定期整理（如每周一次）
- 当出现以下情况时主动建议：
  - 新增了重要项目约定，但不知道该放哪个文件
  - 发现不同文件对同一事情说法不一致
  - 某些 daily 文件已经过时但不确定能否删除
  - MEMORY.md 超过 200 行

## 执行步骤

### Step 1 — 全量扫描

扫描以下所有记忆文件：
```
MEMORY.md                              ← 长记忆层（核心）
memory/YYYY-MM-DD.md                   ← Daily 层（原始日志）
memory/ENTRYPATH.md                    ← 主题索引（如果存在）
```

按时间线汇总所有记忆条目，标记每条记忆的：
- **来源文件**（MEMORY.md 或某个 daily）
- **出现次数**（判断是否重复）
- **最后更新时间**（判断是否过时）
- **置信度**（高：用户明确说过 / 中：推断 / 低：模糊记忆）

### Step 2 — 问题检测

| 问题类型 | 检测逻辑 | 输出标记 |
|----------|---------|---------|
| 重复（Duplicate） | 同一内容在 ≥2 个文件中出现 | 🔁 |
| 过时（Stale） | 旧结论被新 daily 推翻；相对时间描述存在超过7天 | ⏰ |
| 冲突（Conflict） | 同一问题不同文件说法矛盾 | ⚡ |
| 归属模糊（Ambiguous） | 分不清是项目规范还是个人偏好 | ❓ |
| 孤儿条目（Orphan） | 某 daily 中的条目从未被引用或提炼到 MEMORY.md | 🔍 |
| 碎片化（Fragmented） | 一个完整主题分散在 ≥3 个不同文件中 | 📜 |

### Step 3 — 生成提案报告

提案分三类，每一项都必须包含具体内容（旧/新对比）、依据（哪个文件的哪一行）、执行后的预期效果。

## 输出模板（固定格式）

```markdown
## 记忆整理提案

**提案时间**：{{YYYY-MM-DD HH:MM}}
**本次耗时**：{{N}}min
**扫描范围**：MEMORY.md + {{N}}个 daily 文件
**写入位置**：本次无文件写入（仅生成提案，批准后执行）

---

### ▸ 记忆健康度评分

**综合评分：{{N}}/100**

- 入口索引完整度：{{N}}/25
- 内容时效性：{{N}}/25
- 主题聚合程度：{{N}}/25
- 待办跟进率：{{N}}/25

---

### ▸ 问题检测结果

| 类型 | 数量 | 说明 |
|------|------|------|
| 🔁 重复 | {{N}}条 | 同一内容出现在多处 |
| ⏰ 过时 | {{N}}条 | 状态已改变或结论被推翻 |
| ⚡ 冲突 | {{N}}条 | 同一问题不同文件说法矛盾 |
| ❓ 归属模糊 | {{N}}条 | 无法判断是项目规范还是个人偏好 |
| 🔍 孤儿 | {{N}}条 | 从未被引用，价值存疑 |
| 📜 碎片化 | {{N}}条 | 同一主题分散在多处 |

---

### ▸ 必须做(PO)

| # | 行动 | 现状 | 目标 | 说明 |
|---|------|------|------|------|
| P1 | [具体行动] | [当前状态] | [目标状态] | [为什么必须做] |
| P2 | ... | ... | ... | ... |

---

### ▸ 整理清理

| # | 行动 | 现状 | 目标 | 说明 |
|---|------|------|------|------|
| C1 | [归档XX文件] | [文件分散/重复/过时] | [合并到XX/删除] | [解决什么问题] |
| C2 | ... | ... | ... | ... |

---

### ▸ 待确认

以下问题无法自动判断，需要你拍板：

| # | 问题 | A方案 | B方案 | 我的建议 |
|---|------|-------|-------|---------|
| Q1 | [问题描述] | [A] | [B] | [建议] |
| Q2 | ... | ... | ... | ... |

---

**等待批准后执行。** 请回复：
- `同意` → 执行全部行动
- `同意 + 例外 [编号]` → 执行全部，跳过指定条目
- `仅 [编号]` → 只执行指定条目
- `取消` → 不执行任何操作

---
*memory-sorting v1.2.0 · {{YYYY-MM-DD HH:MM}}*
```

## Step 4 — 执行

收到批准后，按以下顺序执行：

```
1. 执行所有 PO（先更新 MEMORY.md，建立新记忆）
2. 执行所有整理清理（删除/合并/更新）
3. 更新 ENTRYPATH.md（如果涉及文件增删）
4. 验证修改后的文件可读
5. 输出执行报告
```

## Step 5 — 执行报告

```markdown
## 记忆整理执行完毕

**执行时间**：{{YYYY-MM-DD HH:MM}}
**实际执行**：{{M}}条（与提案一致 / 有调整）

| 操作 | 数量 | 详情 |
|------|------|------|
| 必须做(PO) | {{N}}条 | [列出] |
| 整理清理 | {{N}}条 | [列出] |
| 跳过 | {{N}}条 | [列出原因] |

**当前 MEMORY.md**：{{N}}行 · 最后更新：{{date}}
**ENTRYPATH.md**：{{N}}条 · 状态：✅已同步
```

## 检测例外（不会标记为问题）

- AI 资讯来源列表（不同 daily 出现相同 URL 是正常现象）
- 完全相同的工具名记录（如"安装了 skill X"，多次出现但上下文不同）
- 时间戳不同但内容相似的 TODO 条目（视为不同时间点的独立记录）

## 权限要求

- `FileRead`：读取所有 memory 文件
- `FileWrite` / `FileEdit`：修改 MEMORY.md 和 daily 文件
- `sessions_spawn`：（可选）fork 子 Agent 并行分析大量文件

- 手动：`/memory-sorting`
