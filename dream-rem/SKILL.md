# dream-rem — 睡梦式记忆深度整合

定时深度整合记忆（默认 ≥24小时 + ≥5个新会话），执行 Orient → Gather → Consolidate → Prune 四阶段整理，将分散的 daily notes 合并到主题索引中，删除过时内容，保持记忆系统长新。

## 何时使用

- **自动触发**：满足 `≥24小时` + `≥5个新会话` 条件时后台自动运行
- **手动触发**：`/dream-rem` 强制触发一次深度整合

## 前置条件

- `extract-memories` 正常工作（daily notes 是整合的原材料）
- `memory/ENTRYPATH.md` 存在（首次运行时会自动创建）
- 写锁 `.consolidate-lock` 不存在（防止多设备同时整理冲突）

## 四阶段执行流程

### ⚙️ Phase 0 — 前置检查

```
检查 .consolidate-lock 是否存在
  → 存在：退出，报告"另有整理在进行"
  → 不存在：创建 .consolidate-lock（写入主机名+时间戳）
扫描 memory/ 目录，确认以下文件存在：
  MEMORY.md
  memory/ENTRYPATH.md
  最近14天内的 daily 文件（N 个）
若 ENTRYPATH.md 不存在：
  → Phase 1 自动升级为"初始化模式"（建立全新索引）
```

### 🔭 Phase 1 — Orient（建立全局视野）

**目标**：理解当前记忆的家底，建立整合坐标

**执行动作**：
1. `ls memory/` 列出所有记忆文件
2. 读取 `memory/ENTRYPATH.md`（主题索引，了解当前知识结构）
3. 扫描 `MEMORY.md`（长记忆层，了解已提炼的精华）
4. 统计文件行数和最后修改时间

**Phase 1 输出模板**：

```markdown
**会话规模**：{{N}}个新会话 · {{总消息数}}条消息

---

**记忆资产总览**
| 层次 | 文件数 | 总行数 | 最新更新 |
|------|--------|--------|----------|
| 入口索引 | 1 | {{N}}行 | {{date}} |
| 长记忆 | 1 | {{N}}行 | {{date}} |
```

### 🔍 Phase 2 — Gather（收集 & 发现问题）

**目标**：从 daily 文件中采集新信息，发现矛盾、过时、重复、空白

**执行动作**：
1. 扫描最近14天的 daily 文件（按时间倒序）
2. 对比 daily 中的信息与 MEMORY.md、ENTRYPATH.md 的差异
3. 检测以下问题类型：
   - **新增（New）**：MEMORY.md 中完全没有的新话题
   - **更新（Update）**：新结论推翻了旧结论
   - **冲突（Conflict）**：同一问题不同地方说法矛盾
   - **重复（Duplicate）**：同一信息在多处出现
   - **过时（Stale）**：描述的状态已改变（如"正在开发"→"已完成"）
   - **升级（Promote）**：零散讨论已形成稳定结论，可提炼到 MEMORY.md

**Phase 2 输出模板**：


### 🧠 Phase 3 — Consolidate（合并 & 整理执行）

**目标**：按 Phase 2 的分析结果，执行实际的整理操作

**执行动作**：
1. 对每条 **Promote** 项：将 daily 中的精华提炼写入 MEMORY.md 或对应主题文件
2. 对每条 **Update** 项：修改 MEMORY.md 中的旧内容为新结论
3. 对每条 **Conflict** 项：与用户确认（输出提案待批准）或直接按建议解决
4. 对每条 **Stale** 项：更新状态描述（相对时间→绝对时间，状态更新）
5. 对每条 **Duplicate** 项：删除重复条目，保留最完整的一个
6. 对每条 **New** 项：创建新的主题文件或更新 ENTRYPATH.md

**Phase 3 输出模板**：

```markdown

| # | 操作类型 | 内容 | 写入位置 |

| ... | ... | ... | ... |

#### ⚠️ 待确认（需用户批准后再执行）

|---|----------|----------|---------|----------|
```

### ✂️ Phase 4 — Prune & Index（精简 & 索引重建）

**目标**：保持记忆系统健康，防止索引膨胀

**执行动作**：
1. 更新 `memory/ENTRYPATH.md`：
   - 确保所有当前活跃主题都有索引条目
   - 删除指向已删除/过时文件的指针
   - 保持每个条目精简（一行标题 + 一行链接）
   - 文件大小控制在 ≤25KB
2. 检查 `MEMORY.md`：
   - 超过 200 行时建议拆分出子主题文件
   - 确保新旧日期格式一致（YYYY-MM-DD）
3. 清理：`consolidate-lock` 文件

**Phase 4 输出模板**：

## 分布式锁机制

为防止多设备同时整理造成冲突：

```
.lock 文件内容：{host}:{pid}:{timestamp}
- 获取锁：尝试创建 .consolidate-lock，写入主机信息
  → 成功：继续执行
  → 失败（已存在）：读取现有锁信息，若超过2小时则强制重获锁，否则退出
- 释放锁：整理完成后删除 .consolidate-lock
- 异常保护：注册 EXIT 钩子，进程结束时确保锁被释放
```

## 权限要求

- `FileRead`：读取所有 memory 文件
- `FileWrite` / `FileEdit`：修改 MEMORY.md、daily 文件、ENTRYPATH.md
- `sessions_spawn`：（可选）fork 子 Agent 并行处理大量文件分析
