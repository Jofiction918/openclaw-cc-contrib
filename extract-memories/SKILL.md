# extract-memories — 对话记忆提炼

每次对话结束后自动分析本轮完整上下文，提炼所有值得持久化的信息，写入 `memory/YYYY-MM-DD.md`，让人工智能在下一次对话时拥有完整记忆，无需用户重复说明背景。

## 何时使用

- **自动触发**：每次对话结束（模型产出最终回复后）自动后台执行
- 手动触发：`/extract-memories` 手动触发一次提炼

## 执行步骤

1. **读取上下文**：获取本轮对话的完整消息历史（sessions_history 或直接引用当前上下文）
2. **全面扫描**：不只关注"明确说出口"的决策，还要挖掘：
   - 用户没有明说但隐含的偏好（语气、格式、沟通风格）
   - 对话中暴露的项目结构和代码位置
   - 涉及的人名、工具名、术语、路径
   - 讨论过程中暴露的空白知识（用户承认不知道的事）
   - 争论的焦点和最终谁说服了谁
3. **多维度分类**：按下方模板的六个维度组织信息
4. **写入 memory/YYYY-MM-DD.md**（APPEND 模式，不覆盖已有内容）
5. **通知主会话**：输出一条简洁确认（提炼了多少条，涉及哪些维度）

## 输出模板

写入文件时严格遵循以下格式，每一项须包含**正文 + 来源依据**（用户原话或具体上下文），方便日后追溯：

## 分类标准说明

**决策（D类）**
- 用户明确同意或选择的方案
- 被否决的替代方案及其否决原因（方便以后避免重蹈覆辙）
- 方向性改变（如"以后这类问题找谁确认"）

**偏好（P类）**
- 沟通风格（正式/随意/技术深度）
- 文档格式偏好（表格 vs 列表 vs 纯文本）
- 工作时间偏好（时区、响应速度期望）
- 工具偏好（明确喜欢或不喜欢某工具）
- 语言偏好（中文/英文/混用）

**约定（C类）**
- 目录结构约定
- Git 工作流（branch 命名规范、commit message 规范）
- 代码风格规范
- 命名规范（变量命名、文件命名）
- 流程规范（review 流程、发布流程）

**技术方案（T类）**
- 问题 → 原因分析 → 解决方案 → 验证方式，完整链路
- 包含具体配置值、版本号、关键代码片段（引用而非粘贴全文）
- 踩坑记录和绕过的坑

**开放问题（Q类）**
- 已知但未解决的问题
- 需要用户或其他成员回答的问题
- 技术债务（知道有更好的方案但暂时没时间做）
- 依赖外部的问题（等别人先完成什么）

**参考资料（R类）**
- 讨论中提到的 URL、文档、论文
- 关键 ChatGPT/AI 对话链接（如果有的话）
- 重要文件路径（尤其是用户没有主动说的暗路径）

## 写入规范

- **APPEND 模式**：追加到 `memory/{{date}}.md` 末尾，不覆盖已有内容
- **文件不存在**：自动创建 `memory/` 目录和文件
- **日期格式**：`YYYY-MM-DD`，使用当天日期
- **标题**：`{{YYYY-MM-DD}} 记忆提炼 · {{本轮核心主题}}`，主题用一句话概括
- **标签**：自动打上相关主题标签，方便后续检索

## 主会话通知格式

提炼完成后，在主会话输出一条确认（不打断用户）：

```
✅ 提炼完成 · {{N}} 条记忆已写入 memory/{{date}}.md
涵盖：{{D类数}}决策 / {{P类数}}偏好 / {{C类数}}约定 / {{T类数}}技术 / {{Q类数}}问题
主题：{{本轮核心主题}}
```

## 权限要求

- `FileRead`：读取对话上下文
- `FileWrite` / `FileEdit`：写入 memory 目录
- `sessions_spawn`：（可选）fork 子 Agent 并行处理大量历史

## 触发词

- 自动：对话结束（skill 内置触发逻辑）
- 手动：`/extract-memories`
